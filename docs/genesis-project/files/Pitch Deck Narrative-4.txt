# I. `$FIGMA_PLUGIN_DATA_SCHEMA`

This schema is the contract a Figma plugin would read and write. Its purpose is to make the deck system **addressable, syncable, and structurally enforceable**, rather than a loose collection of frames. That is consistent with the broader system doctrine: versioned contracts, reusable components, and explicit lineage between abstract system objects and rendered surfaces. fileciteturn2file21 fileciteturn2file24

# IV. INTEGRATION ORDER

```text
1. Store exact copy in $SLIDE_MASTER_CONTENT_MATRIX
2. Bind slides to templates via $FIGMA_PLUGIN_DATA_SCHEMA
3. Bind templates to components via component_registry
4. Bind components to semantic styles via token_bindings
5. Resolve semantic styles through $FRONTEND_THEME_MAP
6. Export Figma and frontend surfaces from the same token + content truth
```

That order preserves the system’s larger law: identity and structure should be versioned and enforced upstream, not improvised downstream. fileciteturn2file23 fileciteturn2file25 fileciteturn2file31

I reviewed the source set. The core document defines the MVP as: one premium asset in, many platform-optimized posts out, then scheduled publishing plus weekly reporting on reach, engagement, and leads. fileciteturn2file0 The later system documents correctly extend that into a research-grade operating system with explicit `assets`, `fragments`, `generated_units`, `publish_events`, and `performance_observations`, and they already sketch the attribution path as `performance → publish_event → unit → fragment → asset`. fileciteturn2file1 The more advanced operator document also makes clear that attribution must not stop at “engagement”; it must connect output to audience movement, economics, and platform state over time. fileciteturn2file2

## Source-grounded conclusion

The mathematically correct attribution model for this system is not single-touch and not purely last-post. It has to be a **hierarchical, time-decayed, multi-touch attribution graph** with five levels:

```text
Asset → Fragment → GeneratedUnit → PublishEvent → PerformanceObservation → AudienceStateTransition / EconomicOutcome
```

That is the only structure consistent with the source documents’ architecture, metrics, and stated goal of turning content generation into an economic engine rather than a posting tool. fileciteturn2file1turn2file2

## Formal attribution model

Let:

```text
a ∈ A                # source asset
f ∈ F(a)             # fragment derived from asset a
u ∈ U(f)             # generated unit derived from fragment f
p ∈ P(u)             # publish event for unit u
o ∈ O(p)             # observation window for publish event p
k ∈ K                # metric type
t                     # time
```

Define base observed metrics per publish event and time window:

```text
y_{p,k,t}
```

Examples:

```text
k ∈ {views, watch_time, engagement, followers_gained, leads, revenue}
```

The source material already implies these as the primary and derived metrics: engagement, followers, leads, asset yield, follow conversion, retention ratio, and ROI. fileciteturn2file0turn2file1turn2file2

## Structural lineage weights

Every publish event inherits from an upstream chain. Define deterministic lineage indicators:

```text
L_asset(a,p)      = 1 if publish event p descends from asset a else 0
L_fragment(f,p)   = 1 if p descends from fragment f else 0
L_unit(u,p)       = 1 if p is generated from unit u else 0
```

If a post is composed from multiple fragments or mixed inputs, replace binary lineage with normalized weights:

```text
w_{a→p} ≥ 0,   Σ_a w_{a→p} = 1
w_{f→p} ≥ 0,   Σ_f w_{f→p} = 1
w_{u→p} ≥ 0,   Σ_u w_{u→p} = 1
```

This is necessary because a carousel, remix, or stitched cut can carry more than one source fragment.

## Time-decayed metric accumulation

A publish event continues to generate value after publication. Let the contribution of publish event `p` to metric `k` over its full life be:

```text
M_{p,k} = ∫_0^T y_{p,k}(τ) · d_k(τ) dτ
```

where `d_k(τ)` is a decay kernel.

A practical discrete version:

```text
M_{p,k} = Σ_{t=0}^{T} y_{p,k,t} · exp(-λ_k t)
```

where:

```text
λ_k > 0
```

Different metrics should use different decay rates. For example, engagement can decay faster than leads; follower gains may decay slower than likes but faster than attributed revenue.

This is aligned with the source demand to model content half-life rather than only first-spike engagement. fileciteturn2file1turn2file2

## Platform-normalized value

Because platform conditions differ, raw metrics must be normalized by platform and time regime. Let:

```text
z_{p,k} = (M_{p,k} - μ_{platform,format,k,era}) / σ_{platform,format,k,era}
```

Then define a platform-adjusted scalar value for a publish event:

```text
V_p = Σ_k α_k · z_{p,k}
```

where `α_k` are strategic weights.

Example:

```text
α_watch_time > α_likes
α_followers_gained > α_comments
α_leads > α_followers when commercial objective dominates
```

This is required because the system documents explicitly treat platform state as time-varying and platform-specific. fileciteturn2file2

## Publish-event attribution to upstream objects

Now attribute publish-event value upward.

### Unit attribution

```text
Attr(u) = Σ_{p ∈ P(u)} w_{u→p} · V_p
```

### Fragment attribution

```text
Attr(f) = Σ_{p} w_{f→p} · V_p
```

### Asset attribution

```text
Attr(a) = Σ_{p} w_{a→p} · V_p
```

This is the direct generalization of the source’s prototype query:

```sql
SELECT asset_id, SUM(engagement)
FROM performance_observations
JOIN publish_events USING (publish_id)
JOIN generated_units USING (unit_id)
GROUP BY asset_id;
```

but generalized to weighted, multi-metric, time-decayed, platform-normalized attribution rather than naive summed engagement. fileciteturn2file1

## Audience-state-transition attribution

The more advanced documents redefine the system around audience formation, not just content output. They specify the audience chain:

```text
Viewer → Engaged Viewer → Follower → Returning Follower → Converted User → Advocate
```

fileciteturn2file2

So define audience states:

```text
S = {0,1,2,3,4,5,6}
```

with transitions:

```text
0→1 unaware→viewer
1→2 viewer→engaged
2→3 engaged→follower
3→4 follower→returning
4→5 returning→converted
5→6 converted→advocate
```

For each publish event `p`, estimate transition counts:

```text
T_{p,i→j}
```

Then define transition value weights:

```text
β_{i→j}
```

and a state-transition value:

```text
Q_p = Σ_{i→j} β_{i→j} · T_{p,i→j}
```

Then total publish-event value becomes:

```text
V_p* = V_p + Q_p
```

This lets attribution model not only post performance, but the actual audience movement the sources identify as the system’s core purpose. fileciteturn2file2

## Economic attribution

Let the cost basis for asset `a` be:

```text
C_a = asset_production_cost + processing_cost + publishing_cost + overhead_allocated
```

Let the asset-attributed economic returns be:

```text
R_a = γ_1 · Attr_followers(a)
    + γ_2 · Attr_leads(a)
    + γ_3 · Attr_revenue(a)
    + γ_4 · Attr_brand_value(a)
```

Then:

```text
GrossAssetReturn(a) = R_a
NetAssetReturn(a)   = R_a - C_a
ROI(a)              = (R_a - C_a) / C_a
AssetYieldMultiplier(a) = R_a / C_a
```

This directly matches the source ambition to move from “beautiful video” to measurable yield, leads, and ROI. fileciteturn2file0turn2file2

## Causal rather than correlational attribution

The source set explicitly says the system must move from correlation to causal inference and treat every post as a micro-experiment. fileciteturn1file0turn2file1

So observational attribution above is necessary but insufficient. Add a causal lift term.

For a treatment variant `r` relative to control `c`, define:

```text
Lift_{k} = E[Y_k | do(r)] - E[Y_k | do(c)]
```

In randomized deployment:

```text
\hat{Lift}_{k} = mean(Y_k | treatment) - mean(Y_k | control)
```

Then fragment-level or unit-level causal value is:

```text
CausalAttr(u,k) = n_u · \hat{Lift}_{u,k}
```

where `n_u` is treatment exposure count.

For non-randomized deployments, use a doubly robust estimate:

```text
DR_k = 1/n Σ_i [ m_1(x_i) - m_0(x_i)
               + z_i (y_i - m_1(x_i))/e(x_i)
               - (1-z_i)(y_i - m_0(x_i))/(1-e(x_i)) ]
```

where:

```text
z_i = treatment indicator
e(x_i) = propensity score
m_1, m_0 = outcome models
```

Then use `DR_k` as the causal increment rather than raw observed delta.

## Natural-center constraint in attribution

The source set repeatedly insists that every output inherit from a Natural Center and that identity be measurable and enforceable. fileciteturn2file2

So attribution should penalize off-brand wins that damage long-term system coherence.

Let:

```text
NCScore_p ∈ [0,1]
```

Then define identity-adjusted event value:

```text
V_p^NC = V_p* · (1 - ρ + ρ·NCScore_p)
```

where `ρ ∈ [0,1]` controls how strongly identity coherence matters.

If `NCScore_p < θ_min`, either reject the unit entirely or discount its attributed value sharply:

```text
V_p^NC = 0    if NCScore_p < θ_min_reject
```

This prevents the model from rewarding generic noise that performs temporarily but erodes brand recognizability.

## Full attribution equation

The cleanest production equation is:

```text
FinalAttr(a) = Σ_p w_{a→p}
               · [ Σ_k α_k · Norm(M_{p,k})
                   + Σ_{i→j} β_{i→j} · T_{p,i→j}
                   + CausalLift_p
                 ]
               · IdentityAdj_p
               · ReliabilityAdj_p
```

where:

```text
Norm(M_{p,k})      = platform/time normalized metric value
CausalLift_p       = experimental or causal increment
IdentityAdj_p      = natural-center coherence modifier
ReliabilityAdj_p   = source reliability modifier for incomplete analytics
```

This is the mathematically correct top-level formula for your system.

## Practical SQL-ready schema logic

At minimum, keep these tables:

```sql
assets(asset_id, brand_id, cost_basis, created_at)

fragments(fragment_id, asset_id, energy_score, semantic_cluster)

generated_units(unit_id, fragment_id, platform, unit_type, nc_version)

publish_events(publish_id, unit_id, platform, published_at, schedule_policy_id)

performance_observations(obs_id, publish_id, observed_at, metric_name, metric_value, reliability)

audience_transitions(transition_id, publish_id, from_state, to_state, transition_count)

attribution_weights(publish_id, asset_id, fragment_id, unit_id, w_asset, w_fragment, w_unit)

experiments(experiment_id, unit_id, control_flag, treatment_flag, hypothesis_id)
```

Then a first non-naive asset attribution view looks like:

```sql
WITH metric_rollup AS (
  SELECT
    po.publish_id,
    SUM(
      CASE po.metric_name
        WHEN 'engagement' THEN po.metric_value * 0.20
        WHEN 'watch_time' THEN po.metric_value * 0.25
        WHEN 'followers_gained' THEN po.metric_value * 0.30
        WHEN 'leads' THEN po.metric_value * 0.25
        ELSE 0
      END * po.reliability
    ) AS metric_score
  FROM performance_observations po
  GROUP BY po.publish_id
),
transition_rollup AS (
  SELECT
    at.publish_id,
    SUM(
      CASE
        WHEN at.from_state = 1 AND at.to_state = 2 THEN at.transition_count * 1
        WHEN at.from_state = 2 AND at.to_state = 3 THEN at.transition_count * 3
        WHEN at.from_state = 3 AND at.to_state = 4 THEN at.transition_count * 4
        WHEN at.from_state = 4 AND at.to_state = 5 THEN at.transition_count * 8
        WHEN at.from_state = 5 AND at.to_state = 6 THEN at.transition_count * 12
        ELSE 0
      END
    ) AS transition_score
  FROM audience_transitions at
  GROUP BY at.publish_id
)
SELECT
  aw.asset_id,
  SUM(
    aw.w_asset * (
      COALESCE(mr.metric_score,0) +
      COALESCE(tr.transition_score,0)
    )
  ) AS attributed_value
FROM attribution_weights aw
LEFT JOIN metric_rollup mr ON mr.publish_id = aw.publish_id
LEFT JOIN transition_rollup tr ON tr.publish_id = aw.publish_id
GROUP BY aw.asset_id;
```

For V1, use explicit priors instead of pretending precision you do not yet have.

A reasonable initialization is:

```text

α_engagement       = 0.15
α_followers_gained = 0.25
α_leads            = 0.35
```

Audience transition weights can start as:

```text

β_3→4 = 4

```

Then update these from empirical outcome data after enough experiments accumulate.
